# 量化交易

## 零、系统定位

| 维度 | 定位 |
| --- | --- |
| 资金规模 | 100 万人民币 |
| 开发模式 | 个人长期开发 |
| 策略频率 | 分钟(主要)、天 |
| 市场范围 | A 股 + 美股 + 港股 |
| 演化周期 | 5 年以上 |
| 系统档位 | 个人长期工程。既不做玩具版,也不上机构重型版。第一版必须能跑通,保留长期扩展能力 |

**核心定位:**

- 不做玩具版,不上机构重型版。
- 外围用成熟框架,核心交易规则自己掌控。
- 第一版先跑通,后面逐步扩展。

## 一、总体框架

### 1.1 总体架构图

```
                  ┌─────────────────────────────────────┐
                  │      Market Adapter (横向枢纽)       │
                  │   cn_stock / hk_stock / us_stock     │
                  │  日历 / 费率 / T+N / 涨跌停 / 最小单位 / 汇率 │
                  └─────────────────────────────────────┘
                                    ↕
       ┌──────────────────────────────────────────────────┐
       │                                                  │
   DataHub ─→ Factor Engine ─→ Strategy Engine ─→ Portfolio + Scheduler
                                                          │
                  ┌───────────────────────┼───────────────────────┐
                  ↓                       ↓                       ↓
          Backtest Engine          Paper Trading            Live Trading
                                          │
                                          ↓
                                     Risk Engine
                                  (事前 / 事中 / 事后)
                                          │
                                          ↓
                                          OMS
                                    (轻量订单管理)
                                          │
                                          ↓
                                    Broker Adapter
                                长桥 / IBKR / 富途 / QMT
                                          │
                                          ↓
                                    Monitor + Audit
                                (实时观察 + 盘后稽核)
                                          │
                                          ↓
                                  Review + Iteration
```

### 1.2 核心设计原则

1. **策略逻辑统一,执行接口分层** —— 一份策略代码,三套市场规则,三套实盘接口
2. **回测、模拟盘、实盘共用同一份策略逻辑** —— 只切换数据源和执行器
3. **市场差异通过轻量 Market Adapter 处理** —— 不做机构级重型抽象
4. **策略只输出目标仓位,不直接下单** —— 这是分层架构的基石
5. **风控、OMS、券商接口严格分层** —— 每层职责单一,不跨层调用
6. **外围用成熟框架,核心规则自己掌控** —— 研究/存储/调度/连接用现成,业务核心自研
7. **先跑通最小闭环,再扩展三市场和多策略** —— 按阶段演化,不一次到位

### 1.3 一句话定位

> 数据进来 → 清洗 → 算指标 → 生成策略目标 → 组合协调 → 回测/模拟/实盘 → 风控 → 下单 → 监控对账 → 复盘优化
>
> **外围用成熟框架,核心交易规则自己掌控**。

## 二、完整流程

### 2.1 简要流程

```
开始
  → 数据采集 DataHub
  → 数据清洗/入库 DataHub
  → 市场规则适配 MarketAdapter
  → 因子计算 FactorEngine
  → 策略生成目标仓位 StrategyEngine
  → 组合管理合并目标 Portfolio + Scheduler
  → 回测验证 Backtest
  → 券商模拟盘 Paper
  → 小资金实盘 Live
  → 风控校验 RiskEngine
  → 订单管理 OMS
  → 分市场执行 BrokerAdapter
  → 监控 / 对账 / 归因 Monitor
  → 策略迭代
```

### 2.2 详细流程说明(小白版)

为了让没接触过量化的人也能看懂,下面用**讲故事的方式**把整条流程串起来。假设你要做一个最简单的动量策略 —— "最近涨得强的股票买入,涨势变弱或跌破规则就卖出",看看从数据到下单到底要经过什么。

---

#### 第一步:采集数据(把"原料"搬回家)

系统先从外部接口拿股票数据。

- 可能来源:
  - 美股:长桥、IBKR、富途、Polygon、yfinance
  - 港股:长桥、IBKR、富途
  - A 股:QMT、Tushare、AKShare
- 拿到的数据包括:
  - 股票代码
  - 时间
  - 开盘价
  - 最高价
  - 最低价
  - 收盘价
  - 成交量
  - 成交额

这些数据就是量化系统的"原材料"。

**数据采集**就是写程序定时去这些地方"取货",存到自己的硬盘上。

> 类比:像每天早上去菜市场把今天要用的食材买回来。
>
> **关键动作**:定时任务自动跑、三市场分别采集、存到统一的 Parquet 文件、失败要重试。

---

#### 第二步:洗菜切配(数据清洗与入库)

刚买回来的菜不能直接下锅,得洗、切、分类。数据也一样,可能有这些问题:

- A 股有**复权问题**:股票分红送股后价格会变,要还原成可比的价格
- 美股有**拆股问题**:苹果一拆四后股价从 600 降到 150,不调整就回测失真
- 港股有**停牌**问题
- 三市场的**字段命名不一样**,要统一(比如开盘价统一叫 `open`)

清洗后的数据存成三层:

- `raw/`:原始数据(没动过的)
- `clean/`:洗干净的(复权后、字段统一)
- `features/`:基础特征(比如 5 日均价、涨跌幅)

> 类比:`raw` 是带泥的菜,`clean` 是洗干净切好的,`features` 是已经调味腌制好的半成品。

---

#### 第三步:认识规则(市场规则适配)

A 股、港股、美股规则差异巨大:

- A 股**今天买的明天才能卖**(T+1),港美股可以当天买卖(T+0)
- A 股有 **±10% 涨跌停**,港股美股没有
- A 股最少买 **100 股**,港股**每只股票每手数量不一**(腾讯每手 100、汇丰每手 400),美股 **1 股**就能买
- 三市场用**不同货币**

**Market Adapter** 就是个"翻译器":策略只需要说"我要给苹果加仓 10%",Adapter 会自动算出"按美元 1 股最小单位、按当前汇率折算成人民币、检查交易时段是否允许"。

> 类比:你跟司机说"去机场",司机会根据你在北京还是香港,自动选高速、选路线。你不需要告诉司机交通规则。

---

#### 第四步:加工原料(因子计算)

因子是策略的"半成品零件"。比如:

- **动量因子**:过去 20 分钟涨幅
- **价值因子**:市盈率(PE)
- **波动率因子**:过去 30 分钟价格波动幅度

```
AAPL    过去 60 分钟涨幅 =  2.1%
腾讯    过去 60 分钟涨幅 =  1.3%
茅台    过去 60 分钟涨幅 = -0.5%
```

这些就是策略判断的依据。算出来后存到 `factors/` 带版本号(如 `momentum_20m_v1`),下次直接读缓存,不用重算 —— 分钟频策略每天会跑很多次,不缓存会爆机。

> 类比:餐厅不会每次客人点菜才去切配料,而是每天早上批量切好放冰箱。

---

#### 第五步:决定吃什么(策略生成目标仓位)

策略读取因子,根据规则输出**目标仓位**:

```python
{
    "AAPL":      0.10,   # 苹果加仓到 10%
    "00700.HK":  0.08,   # 腾讯减到 8%
    "600519.SH": 0.00,   # 茅台清仓
}
```

**关键铁律**:策略**不允许直接调用券商 API 下单**,只能输出"我希望最终账户长什么样",而不是"现在马上买 100 股"。

> 类比:你跟厨师说"我要吃番茄炒蛋",但具体怎么炒、火候多大、什么时候下盐,是厨师(后面的执行模块)的事。
>
> 为什么这么严格?因为这样**回测和实盘可以共用一份策略代码** —— 回测时是"模拟厨师",实盘时是"真实厨师",但菜谱(策略代码)只有一份。

---

#### 第六步:统一意见(组合管理)

如果你同时跑了多个策略,可能会出现矛盾:

- 策略 A 说苹果买 10%
- 策略 B 说苹果卖 5%

**Portfolio 模块**负责协调,按规则合并(如按权重加权、按优先级覆盖)。还做**全局约束**:单票 < 15%、单市场 < 50%、总杠杆 < 1 倍。

> 类比:家里几个孩子各自报需求,家长根据时间和钱包统一决定今天去哪。

---

#### 第七步:停车场试驾(回测验证)

把策略放到过去 3-5 年的历史数据上跑一遍,看赚不赚钱、最大亏多少。**回测必须模拟真实交易摩擦**:手续费、印花税、滑点、T+1、涨跌停、停牌、最小交易单位、汇率。

判断能不能上实盘的标准(按策略类型分档):

| 策略类型 | 进入实盘门槛 |
| --- | --- |
| 趋势/动量 | Sharpe > 0.8,最大回撤 < 25% |
| 多因子选股 | Sharpe > 1.0-1.2,最大回撤 < 25% |
| 套利/对冲 | Sharpe > 1.5-2.0,最大回撤 < 10% |

> 类比:开新车前先在停车场练几圈,看刹车灵不灵、油门稳不稳,再上路。

---

#### 第八步:正式彩排(券商模拟盘)

回测过了不代表能直接上真金白银。要先用**券商模拟账户**(长桥 Paper、IBKR Paper、QMT 模拟)跑一段时间,验证 API 接口、断线重连、订单回报、实际成交价。

模拟盘**不是验证策略赚不赚钱**(那是回测的事),**而是验证系统能不能正常执行**。至少跑 2-4 周,确认没大问题再上小资金。

> 类比:演员排练完剧本,还要在真实舞台上彩排一次,看灯光、走位、麦克风,才正式演出。

---

#### 第九步:小心试水(小资金实盘)

模拟盘跑通后,先用**小资金实盘**(比如 5-10 万),目的不是马上赚钱,而是验证真实滑点、真实成交率、真实网络稳定性、真实风控效果。

跑满 1-3 个月,绩效与回测偏离可控,再放大资金。

---

#### 第十步:安检通道(风控校验)

每笔订单在真正发出之前,必须过三道关卡:

- **事前(下单前)** :仓位是否超限、现金是否够、是否重复下单、是否停牌涨跌停
- **事中(运行中)** :今日亏损是否超 3%、回撤是否超 15%、异常成交报警
- **事后(收盘后)** :持仓对账、资金对账、订单对账

> 类比:安检员在乘客上飞机前查一次(事前)、空乘在飞行中观察(事中)、地勤在落地后清点货物(事后)。

---

#### 第十一步:订单派件(OMS 订单管理)

OMS 把策略想要的"目标仓位"翻译成具体的"买入/卖出订单",并追踪每个订单状态:下单成功了吗?部分成交了多少?撤单成功了吗?超时怎么办?

**重要边界**:OMS 不自己拆单,拆单交给券商算法订单(IBKR 的 TWAP、长桥的 ALGO)。

> 类比:OMS 是快递站的派件员,负责确认每个包裹是不是真的发出去了,但他不开货车 —— 开货车是物流公司的事。

---

#### 第十二步:跨国邮政(分市场执行)

Broker Adapter 是最后一公里:把统一格式的订单翻译成每家券商能听懂的语言。

- A 股 → 调用 QMT 的 XtQuant API
- 港股 + 美股 → 调用长桥或 IBKR 的 API

> 类比:你想跨国寄包裹,不需要懂每个国家的邮政系统,DHL 帮你统一处理。

---

#### 第十三步:看清仪表盘 + 每天对账(监控、对账、归因)

实盘运行起来,**Monitor** 实时观察系统(实时 PnL、持仓、报警),**Audit** 每天收盘后做对账:

- **持仓对账**:系统持仓 vs 券商持仓,差 > 1 股报警
- **资金对账**:系统记账 vs 券商资金,差 > 1 元报警
- **订单对账**:核对漏单错单
- **滑点分析**:实际成交 vs 回测假设

港美股建议**双记账**:人民币口径 + 原币种口径,避免汇率波动导致对账失败。

> 类比:开店要看每天的流水账(Monitor),还要每月盘点库存(Audit),才知道生意好不好、有没有损耗。

---

#### 第十四步:持续进化(策略迭代)

量化是个**持续工程**:因子会衰减、市场会变化、滑点会失控。每周/每月做一次复盘,看哪个策略要调整、要加新因子、要换市场。

> 类比:种地不是播种就完事,还要除草、施肥、轮作,常年累月地照看。

---

### 2.3 一句话总结

> **数据从外面进来 → 经过清洗、加工、抽象、计算 → 变成策略的目标仓位 → 经过组合协调、回测验证、模拟演练 → 经过风控安检 → 由 OMS 派发到对应市场的券商 → 实盘运行 → 每日对账 → 持续监控和优化**。
>
> 整个链路上,**策略逻辑只写一次**,前面接什么数据源、后面接什么券商,通过 Adapter 适配。

## 三、核心模块详解

每个模块按 **定义 → 功能 → 边界 → 上下游交互 → 技术框架选择** 五段展开。

---

### 3.1 DataHub 数据模块

**定义**:整个系统的"数据总管",负责把外部分散的市场数据统一采集、清洗、存储,并向系统其他模块提供标准化的数据访问接口。

**功能说明**:

- 三市场行情采集:分钟线、日线、Tick(可选)
- 基本面数据:财报、市值、行业分类
- 复权与拆股调整:A 股前复权/后复权,美股 split adjustment
- 停牌、涨跌停标记
- 汇率数据(美元/港币/人民币)
- 数据质量校验(缺失检测、异常值检测)
- 提供统一查询接口

**数据分层**(起步 3 层,长期预留 6 层):

```
raw/        # 原始数据,完全不动
clean/      # 复权、字段统一后的标准数据
features/   # 基础衍生(均线、涨跌幅)
factors/    # 复合因子(后期加)
signals/    # 策略信号历史(后期加)
positions/  # 持仓快照(后期加)
```

**边界说明**:

- ✅ 负责数据"取得"和"存储",**不做策略相关计算**
- ✅ 提供标准接口,**不关心调用方是回测还是实盘**
- ❌ 不做因子计算(那是 Factor Engine 的事)
- ❌ 不做交易日历判断(那是 Market Adapter 的事)
- ❌ 不存订单/持仓/账户状态(那是 OMS + PostgreSQL)

**上下游交互**:

- **上游**:Tushare、AKShare、长桥行情 API、IBKR 行情、Polygon、yfinance
- **下游**:Factor Engine、Backtest Engine、Strategy Engine、Market Adapter、Monitor

**技术框架选择**:

- **存储**:Parquet(历史行情,按市场/年/月分区)+ DuckDB(研究查询)+ PostgreSQL(元数据)
- **采集**:`tushare` / `akshare` / 各券商 SDK
- **计算**:`pandas` 起步,后期换 `polars`(快 5-10 倍)
- **质量校验**:`pandera`(可选)
- **判断**:**半自研** —— 存储和计算用成熟工具,采集和清洗逻辑自写
- **不上**:ClickHouse(数据 < 500GB 不需要)、Redis(实时状态需求出现再加)

---

### 3.2 Market Adapter 市场适配层

**定义**:三市场差异的统一抽象层。把 A 股、港股、美股各自的"规则差异"封装成统一接口,让上层模块不需要关心市场细节。

**功能说明**:

| 抽象项 | A 股 | 港股 | 美股 |
| --- | --- | --- | --- |
| 交易日历 | 沪深交易所 | 港交所 | NYSE/NASDAQ |
| 交易时段 | 9:30-11:30, 13:00-15:00 | 9:30-12:00, 13:00-16:00 | 9:30-16:00 |
| 费率 | 佣金 + 印花税 0.1% + 过户费 | 佣金 + 印花税 + 交易费 | 佣金 + SEC fee |
| T+N | T+1 | T+0 | T+0 |
| 涨跌停 | ±10%/20%/30% | 无 | 熔断机制 |
| 最小交易单位 | 100 股 | 每股不一 | 1 股 |
| 最小变动价位 | 0.01 元 | 按价位分档 | 0.01 美元 |
| 币种 | CNY | HKD | USD |

**核心接口**:

```python
class MarketAdapter:
    def is_trading_day(date) -> bool
    def get_trading_hours(date) -> tuple
    def get_lot_size(symbol) -> int
    def get_tick_size(symbol, price) -> float
    def get_fee(symbol, price, quantity, side) -> float
    def is_tradable(symbol, time) -> bool
    def round_price(symbol, price) -> float
    def round_quantity(symbol, qty) -> int
```

**边界说明**:

- ✅ 抽象**90% 策略都会用到的**市场规则
- ❌ **不抽象**特殊订单类型、券商私有功能、算法订单细节
- ❌ 不存数据,只做规则判断
- 判断标准:**90% 策略都要处理的差异 → 抽象;只有 10% 用到 → 留在各市场实现**

**上下游交互**(横向枢纽,几乎所有模块都会用):

- **上游**:`market.yaml` 配置、DataHub、汇率数据、交易所日历
- **下游**:Factor Engine、Strategy Engine、Backtest Engine、Portfolio、Risk Engine、OMS、Broker Adapter

**技术框架选择**:

- **交易日历**:`exchange_calendars`(覆盖全球 60+ 交易所,**强烈推荐**,A/港/美三市场日历它全有)
- **时区处理**:`pytz` 或 `zoneinfo`
- **汇率**:各券商 SDK 自带,或外部接口
- **判断**:**必须自研核心骨架** —— 这是系统能否长期演化的关键
- **目录**:

  ```
  market/
    base.py       # 抽象接口
    cn_stock.py   # T+1, ±10%, 100股, 印花税
    hk_stock.py   # T+0, 无涨跌停, 每手不一, 港币
    us_stock.py   # T+0, 熔断, 1股, 美元
  ```

---

### 3.3 Factor Engine 因子引擎

**定义**:从清洗后的数据中计算各种"特征值"(因子),供策略使用,并管理因子的版本与缓存。

**功能说明**:

- 技术因子:动量、均线、波动率、RSI、MACD……
- 基本面因子:PE、PB、ROE、净利润增长率
- 量价因子:换手率、成交额比、价量背离
- 跨市场因子:A/H 溢价、ADR 折溢价
- **因子版本管理**(改公式后旧因子还能查得到)
- **因子缓存**(分钟频每天跑很多次,不能每次重算)
- **因子有效性分析**(IC、IR、分层回测)

**因子版本管理示例**:

```python
factor_meta = {
    "id": "momentum_20m_v2",
    "formula": "close.pct_change(20).rolling(60).mean()",
    "created_at": "2026-01-15",
    "deprecated": False,
}
# 元数据存 PostgreSQL,数据按 factor_id 存 Parquet
```

**边界说明**:

- ✅ 负责**特征工程**和**因子库管理**
- ❌ 不做策略决策(因子只是"原料",怎么用是 Strategy 的事)
- ❌ 不直接读外部数据(只读 DataHub 的 clean/features 层)
- ❌ 不调用券商 API

**上下游交互**:

- **上游**:DataHub(读 clean 和 features 层)、Market Adapter(交易日历)
- **下游**:Strategy Engine(读因子值)、Backtest Engine、Monitor(展示因子有效性)

**技术框架选择**:

- **向量化计算**:`pandas` 起步,后期换 `polars`
- **技术指标**:`pandas-ta`(纯 Python,装起来不踩坑) / `TA-Lib`(C 实现,后期性能瓶颈时用)
- **因子检验**:`alphalens-reloaded`(IC/IR/分层回测,Quantopian 留下的好东西)
- **判断**:**半自研** —— 计算用成熟库,因子管理与版本控制自写
- **不上**:微软 Qlib(过重,学习成本高,不适合个人起步)

### 3.4 Strategy Engine 策略引擎

**定义**:量化系统的"大脑"。读取因子和行情,根据策略逻辑输出**目标仓位**。

**功能说明**:

- 从 Factor Engine 读因子
- 从 DataHub 读最新行情
- 通过 Market Adapter 检查标的可交易性
- 执行策略规则(动量、均线、轮动、套利等)
- 控制调仓频率
- 输出标准化的目标仓位列表

**正确输出格式**:

```python
[
    {"symbol": "AAPL",      "market": "US", "target_weight": 0.10},
    {"symbol": "00700.HK",  "market": "HK", "target_weight": 0.08},
    {"symbol": "600519.SH", "market": "CN", "target_weight": 0.00},
]
```

**核心铁律**:

- ✅ 策略**只输出目标**,不下单
- ✅ 策略**不知道**自己在回测、模拟还是实盘
- ❌ 策略里**绝对不能写** `broker.place_order(...)`
- ❌ 策略里**不应该有**"如果是实盘就 X,如果是回测就 Y"的分支判断

**边界说明**:

- ✅ 只关心"应该持有什么",不关心"怎么交易到那个仓位"
- ❌ 不做组合层面的协调(那是 Portfolio 的事)
- ❌ 不做风控校验(那是 Risk Engine 的事)

**上下游交互**:

- **上游**:DataHub、Factor Engine、Market Adapter
- **下游**:Portfolio(把目标仓位提交给组合管理)、Backtest Engine、Paper Trading、Live Trading

**技术框架选择**:

- **完全无现成框架可用** —— 这是你的 alpha 所在
- **判断**:**必须自研**
- **设计要点**:定义统一的策略基类(StrategyBase),所有策略继承并实现 `generate_targets()` 方法
- **辅助库**:`pandas` / `polars`(数据处理)、`numpy`(数值计算)

---

### 3.5 Portfolio 组合管理模块

**定义**:多策略、多市场的仓位协调与资金分配中枢。

**功能说明**:

- **多策略目标合并**:多个策略对同一标的有不同目标时按规则合并
- **资金分配**:每个策略分配多少资金
- **全局约束**:
  - 单票最大权重(如 15%)
  - 单市场最大权重(如 50%)
  - 单策略最大资金占用
  - 行业暴露限制
- **再平衡**:把当前持仓和目标仓位的差值算出来,生成调仓订单

**示例**:

```
策略 A 输出:AAPL 目标 10%
策略 B 输出:AAPL 目标 -5%

Portfolio 处理:按权重 0.6/0.4 加权 → AAPL 最终 4%
                + 全局约束:AAPL 不超过 15%,4% 通过

当前持仓:AAPL 7%
生成订单:卖出 AAPL 3%
```

**边界说明**:

- ✅ 做**目标层面的协调**和**全局约束检查**
- ❌ 不做事前/事中风控(那是 Risk Engine 的事,Risk 比 Portfolio 更底层)
- ❌ 不直接下单(订单交给 OMS)

**上下游交互**:

- **上游**:Strategy Engine(收集多策略目标)、Risk 配置、Market Adapter
- **下游**:Risk Engine(订单送去校验)、Backtest/Paper/Live(用合并后的目标驱动)

**技术框架选择**:

- **判断**:**必须自研轻量版** —— 业务规则强相关,无现成框架
- **辅助库**:`PyPortfolioOpt`(可选,做组合优化、马科维茨、风险平价时用)
- **数值计算**:`numpy` / `cvxpy`(凸优化,做风险约束时用)

### 3.6 Scheduler 调度模块

**定义**:任务调度器,负责定时触发各种作业。

**功能说明**:

- 定时拉数据(盘前、盘中、盘后)
- 定时算因子
- 定时跑策略(分钟频每分钟跑一次)
- 定时下单
- 定时对账
- 定时生成报表

**边界说明**:

- ✅ 只做"什么时间触发什么任务"
- ❌ 不实现具体业务逻辑

**上下游交互**:被调度的对象包括 DataHub、Factor、Strategy、Audit 等所有需要定时跑的模块。

**技术框架选择**:

- **起步**:`APScheduler`(Python 调度事实标准,够用)
- **后期**:任务依赖复杂或跨机器后,升级到 `Prefect` 或 `Airflow`
- **判断**:**完全不自研**,直接用现成框架
- **示例**:

  ```python
  @scheduler.scheduled_job(cron='*/5 9-15 * * 1-5')
  def run_strategy():
      ...
  ```

### 3.7 Backtest Engine 回测引擎

**定义**:在历史数据上模拟运行策略,评估策略表现。**与实盘共用 Strategy 代码**,只是把"真实下单"换成"模拟撮合"。

**功能说明**:

- 历史数据回放(按时间循环喂数据)
- 模拟撮合(订单按规则成交)
- 费用计算(手续费、印花税、滑点)
- 持仓和资金更新
- 绩效统计(收益、夏普、回撤、胜率)

**架构原则**:

```
一份策略逻辑(与实盘共用)
   +
一套统一撮合主流程
   +
三套市场规则配置(从 Market Adapter 读)
```

**撮合主流程**:

```
订单进入 → 市场规则校验(涨跌停/停牌/T+1)
       → 撮合成交(按当根 K 线模拟价格)
       → 费用计算
       → 持仓更新
       → 资金更新
```

**必须模拟**:手续费、印花税、滑点、T+1、涨跌停、停牌、最小交易单位、汇率、交易时段。

**边界说明**:

- ✅ 撮合**主流程统一**,**规则参数化**
- ✅ **特殊机制插件化**(A 股 T+1 用插件实现,美股不加载)
- ❌ 不写"三套独立回测引擎"
- ❌ 不写"万能引擎"
- ❌ 不模拟微观结构(分钟级回测不需要订单簿)

**上下游交互**:

- **上游**:DataHub(历史数据)、Strategy Engine(策略代码)、Market Adapter(规则)
- **下游**:Monitor(回测报告)、Review(策略迭代决策)

**技术框架选择**(三阶段演进,关键设计):

| 阶段 | 工具 | 用途 |
| --- | --- | --- |
| 阶段 1(0-3 月) | `vectorbt` | 起步快速验证简单动量,只做美股,1 周能跑回测,参数扫描极快 |
| 阶段 2(3-9 月) | 自研轻量引擎(polars) | 扩展三市场,300-500 行代码,精准模拟 T+1 / 涨跌停 / 每手不一 |
| 阶段 3(9 月+) | 两者并存 | vectorbt 跑参数扫描和因子研究,自研引擎做准实盘回测 |

**为什么不用 Backtrader**:更适合事件驱动单市场,扩展三市场规则要 hack 核心代码;**vnpy 偏期货 CTA**,不适合多市场股票;**Zipline 已停止维护**。

**辅助库**:`empyrical-reloaded`(夏普、最大回撤、卡玛比率等十几个风险指标)。

**判断**:**先框架后自研** —— 阶段 1 用 vectorbt,阶段 2 自研,阶段 3 研究/生产分离。

### 3.8 Paper Trading 模拟盘模块

**定义**:连接真实模拟账户,验证执行链路的完整性。

**功能说明**:三层递进

| 层 | 形式 | 验证什么 |
| --- | --- | --- |
| 1 | 本地 Paper Broker | 策略逻辑、订单格式 |
| 2 | **券商模拟账户** | API 接口、成交回报、断线重连 |
| 3 | 小资金实盘 | 真实滑点、成交率、市场冲击 |

**铁律**:**分钟频策略不能只做第 1 层**,必须跑第 2 层至少 2-4 周才能上第 3 层。

**边界说明**:

- ✅ 只做执行链路验证,不修改策略
- ❌ 不替代真实回测(回测验证策略是否赚钱,Paper 验证系统是否正常工作)

**上下游交互**:与 Live Trading 路径相同,只是 Broker Adapter 指向模拟账户。

**技术框架选择**:

- **券商模拟账户**:**直接用券商提供的** —— 长桥 Paper(免费、基于真实行情)、IBKR Paper Account(免费,与主账户绑定)、富途模拟、QMT 模拟
- **本地 Paper Broker**:**自研轻量版**(简化版撮合引擎)
- **判断**:**不自研为主** —— 主要靠券商提供的能力,只在最简单层级自研

### 3.9 Risk Engine 风控引擎

**定义**:整个系统的"安检系统",在订单流转的多个环节做安全校验。

**功能说明(三层风控)** :

**事前(下单前校验)** :

- 单票最大仓位(如 15%)
- 单市场最大仓位(如 50%)
- 单策略最大仓位
- 现金充足校验
- 重复下单保护
- 涨跌停/停牌过滤
- 异常价格检查(限价单偏离市价 ±5% 拦截)

**事中(运行中监控)** :

- 单日最大亏损(如 -3%)→ 暂停交易
- 最大回撤熔断(如 -15%)→ 全部清仓
- 异常成交报警
- 持仓敞口监控(单一行业 < 30%)
- 断线报警

**事后**:对账归到 Audit 模块(由 Audit 提供数据,Risk 决定是否熔断)

**边界说明**:

- ✅ 风控是**最后防线**,事前在 OMS 之前,事中独立运行
- ❌ 不替代 Portfolio 的全局约束(Portfolio 关注"目标合理性",Risk 关注"执行安全性")
- ❌ 不做策略层面判断(不评判策略好坏)

**上下游交互**:

- **事前**:接收 Portfolio 的订单 → 校验 → 通过给 OMS
- **事中**:监听实时 PnL 和持仓,有问题时**直接调用 OMS 撤单/平仓**
- **事后**:从 Audit 拿对账结果,决定是否触发熔断

**技术框架选择**:

- **判断**:**必须自研** —— 业务规则强相关,无现成框架
- **风险指标计算**:`empyrical-reloaded`(夏普、回撤、卡玛、索提诺等)
- **组合优化**:`cvxpy`(凸优化,做风险约束时用)
- **设计要点**:规则配置化,所有阈值放在 `risk.yaml`,代码读配置不写死

### 3.10 OMS 订单管理模块

**定义**:订单生命周期的管理者,把"目标仓位差"翻译成具体订单,并追踪每个订单的状态。

**功能说明**:

- 订单生成(从目标仓位差计算订单)
- 订单编号(全局唯一 ID)
- 状态追踪(已发送/部分成交/全部成交/已撤销/已失败)
- 撤单
- 失败重试(网络错误时退避重试)
- 断线恢复(重启后能恢复未完成订单状态)
- 成交记录入库
- 持仓同步

**边界说明**:

- ✅ 做**轻量订单状态管理**
- ❌ **不自研**智能拆单(TWAP/VWAP/盘口拆单)
- ❌ **不做**高频路由(交给券商)
- ❌ 不做风控(那是 Risk 的事)

**自研拆单的触发条件**:单票下单量 > 该票日均成交额 3-5%,**当前阶段不需要**,直接用券商算法订单。

**上下游交互**:

- **上游**:Risk Engine 通过校验后的订单
- **下游**:Broker Adapter(发出真实订单)、PostgreSQL(订单状态持久化)、Monitor(订单日志)、Audit(对账数据)

**技术框架选择**:

- **判断**:**必须自研轻量状态机** —— 订单状态机、对账逻辑、断线恢复必须严格定制
- **持久化**:PostgreSQL(订单/成交/持仓表)
- **状态机**:用 Python 字典或 enum 实现,200-300 行代码
- **不要用**:vnpy 的 OMS(深度耦合 vnpy 全家桶)

### 3.11 Broker Adapter 券商接口模块

**定义**:把统一格式的订单翻译成各家券商 API 能识别的具体调用。

**功能说明**:

- 统一接口对外暴露:

  ```python
  place_order(symbol, side, quantity, order_type, price=None, ...)
  cancel_order(order_id)
  get_orders()
  get_trades()
  get_positions()
  get_cash()
  ```

- 各市场底层实现各自的 SDK 调用

**推荐选择**(明确演进路径):

| 市场 | 起步 | 长期主力 | 补充 |
| --- | --- | --- | --- |
| 美股 | **长桥** | **IBKR + ib_insync** | 富途 |
| 港股 | **长桥** | IBKR | 富途 |
| A 股 | **国信证券 + QMT** | QMT(主) | PTrade(备) |

**演进逻辑**:

- **长桥起步**:无本地网关、实时行情免费、内地身份开户最顺
- **IBKR 长期**:全球覆盖、合规最强、专业级,适合资产规模上来后
- **QMT/PTrade**:A 股唯一选择

**边界说明**:

- ✅ 只做 API 翻译和连接管理
- ❌ 不做订单状态管理(那是 OMS 的事)
- ❌ 不做风控
- ❌ 不缓存数据
- **核心铁律**:**只有 Broker Adapter 触碰真实券商 API**,所有其他模块通过它访问

**上下游交互**:

- **上游**:OMS、Risk Engine、Audit
- **下游**:长桥 API、IBKR API、富途 OpenAPI、QMT API

**技术框架选择**:

- **长桥**:官方 `longport` SDK(7 种语言,Python 友好)
- **IBKR**:`ib_insync`(社区维护,比官方 TWS API 易用 10 倍)
- **富途**:官方 `futu-api` + 本地 OpenD 网关
- **QMT**:`xtquant`(QMT 官方,通过券商 QMT 客户端安装)
- **判断**:**半自研** —— SDK 用现成,Adapter 包装层自写(200-400 行)
- **目录**:

  ```
  brokers/
    base.py         # 统一接口
    longbridge.py   # 长桥实现
    ibkr.py         # IBKR 实现
    futu.py         # 富途实现
    qmt.py          # QMT 实现
  ```

### 3.12 Monitor + Audit 监控稽核模块

**定义**:Monitor 负责实时观察系统状态,Audit 负责盘后检查账是否对得上。两者职责互补,合并为一个模块。

**功能说明**:

**Monitor 实时观察**:

- 策略运行日志、数据任务日志、订单日志、成交日志
- 实时 PnL(今日盈亏、累计、按策略/市场分解)
- 持仓快照
- 风险指标(回撤、敞口、波动率)
- 异常报警(实时推送)
- 接口延迟监控

**Audit 盘后稽核**:

- **持仓对账**:系统持仓 vs 券商持仓,差 > 1 股报警
- **资金对账**:系统记账 vs 券商资金,差 > 1 元报警
- **订单对账**:核对漏单、重复成交、状态不一致
- **滑点分析**:实际成交价 vs 回测假设
- **成交率分析**
- **盘后日报、周度归因**

**港美股双记账**:人民币口径(总资产视角) + 原币种口径(实际交易视角),避免汇率波动导致对账失败。

**职责协同**:

- **Audit**:发现账不平,提供对账报告
- **Monitor**:实时报警和展示
- **Risk**:决定是否停机、熔断、禁止交易(基于 Audit 的报告)

**边界说明**:

- ✅ Monitor 只做**观测**和**报警**,不参与决策
- ✅ Audit 只做**事后稽核**和报告生成,不修改策略
- ❌ 不做事前/事中风控
- ❌ 不直接执行熔断(那是 Risk 的事)

**上下游交互**:

- **上游**:所有模块、Broker Adapter、OMS、Risk Engine、PostgreSQL
- **下游**:日报、报警(飞书/Telegram/邮件)、复盘、策略迭代

**技术框架选择**:

- **日志**:`loguru`(比标准 logging 易用 10 倍)
- **时序指标**:`prometheus_client`
- **可视化**:**起步用文件日志 + 飞书机器人**,实盘后上 `Grafana`
- **报警**:飞书 / Telegram bot(用 webhook,自己写 50 行包装)
- **结构化数据**:PostgreSQL(订单/成交/持仓/对账记录)
- **判断**:
  - Monitor:**不自研为主**(用 loguru/Grafana)
  - Audit:**必须自研**(对账规则与账户结构强相关,SQL + 自定义规则)
